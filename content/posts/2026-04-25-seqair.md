I've been working on [Rastair],
a bioinformatics tool for variant and methylation calling,
for a while now.
I previously wrote about it [here][rastair-post]
and about its GPU-based Random Forest inference [here][ml-post].
One of the big aspects of this is reading and writing files
in different formats
in efficient and correct ways.

[Rastair]: https://www.rastair.com/ "Rastair website"
[rastair-post]: https://deterministic.space/rastair.html "Notes on Rastair, a variant and methylation caller"
[ml-post]: https://deterministic.space/gpu-random-forest-ml.html "Random Forest ML on GPU"

## Formats

The main file formats we deal with are:

FASTA
:  Stores a genomic sequence, we use it for the reference.

BGZF
:  Block-wise gzip compression, which allows indexing and decompression segments of a file.

SAM (sequence alignment map)
:  Main Input for Rastair.
:  Genome sequences/DNA snippets, with lots of metadata.

BAM
:  Binary version of SAM, very common.

CRAM
:  Complex more compressed version.

VCF (variant call file)
:  Main output for Rastair.
:  List of positions and their variants and other information.

BCF
:  Binary version of VCF

There are also matching index files (FAI, GZI, BAI, etc.)
that allow direct indexing.

SAM, VCF, and other formats are specified and maintained
by the team behind [samtools]
and is implemented in their library [`htslib`].
It also implements many other tools and features
and is what I would call the default choice.

There is a Rust wrapper, aptly named [`rust-htslib`],
that Rastair uses.
It is quite decent
but doesn't hide the fact that it wraps a C library.
As of writing this (April 2026),
it also trails behind the latest htslib release[^hts-fork].

[`htslib`]: https://github.com/samtools/htslib "samtools/htslib: C library for high-throughput sequencing data formats"
[samtools]: https://www.htslib.org/ "Samtools"
[`rust-htslib`]: https://github.com/rust-bio/rust-htslib "rust-bio/rust-htslib: This library provides HTSlib bindings and a high level Rust API for reading and writing BAM files."
[`noodles`]: https://docs.rs/noodles/0.109.0/noodles/ "Bioinformatics I/O libraries in Rust"

[^hts-fork]: I forked both `rust-htslib` and `hts-sys` to update both the bindings and add some features.
  My PRs aren't merged yet and there is more work I did that I could upstream.

There is a Rust-native alternative called [`noodles`]
which seems to cover a broad area of these formats.
In my testing however,
it seemed slower than `htslib`
and even produced output that I could not get to work.
It looks like a great effort, though!
It surely deserves more attention (and contributions) than I'm giving it here.

## An experiment

I was very intruiged by the challenge of all of this
and also by trying to understand the formats and their uses on a deeper level.
Trying to optimize BCF writing in `rust-htslib` was how I got started,
and I didn't want to stop there.

It was the week before Easter and we just released Rastair 2.1.
The next big task I want to take on is local read realignment[^realignment]
which requires changing the order and annotations of BAM records in memory.
I knew that for what Rastair needs there are several possible implementations
and htslib and noodles are each presenting only one of them.
I had a vague idea of how I'd want to do it.

[^realignment]: With our knowledge of how TAPS methylation looks,
  we think we fix some alignment issues in regions with a lot of insertions or deletions.

With this, I decided to do an experiment.
Can I write my ideas as specifications using [tracey] (I've heard good things),
combine this with the the extensive specifications for the formats
and use Claude Code to get a prototype for this going?
The goal is to prove whether my ideas are feasible
and also have some fun with trying out new tools
and see how they deal with some of my coding style choices.

[tracey]: https://tracey.bearcove.eu/ "Spec coverage for code"

The prototype is done when I have
a library that can replace `rust-htslib` for the pileup[^pileup] generation
that Rastair does
and show an example of how realignment would be handled.

[^pileup]: We have many reads that are short overlapping snippets of DNA.
  You can imagine this as a 2D grid with the position on the x axis
  and the reads as a stack going up (or down).
  A "pileup" is the column view, where each column shows all the bases for a position.
  In practice, there is also a lot of metadata.

### Side quest: Getting solid code from LLMs

Over the last couple years,
I've several tried LLMs and coding tools based on them.
My success rate was so-so overall.
The first of its kind that I used was "Tabnine" around 2019,
and I got access to Github Copilot at some point but didn't make much use of it.
End of 2025,
I gave Claude Code a try,
and was impressed by what I could get out of it,
given enough hand-holding.
To me, the issue was always the same:
If you know what you're doing,
and you spend some time writing out your thoughts[^rubber],
then it works great.
If you skip this part and hope for the best…
then I guess
"the next generation of models will be so much better"
is what you hear a lot.

[^rubber]: It's the classic [rubber duck] approach,
  where describing your problems and your thoughts
  leads you to the solution.

[rubber duck]: https://en.wikipedia.org/wiki/Rubber_duck_debugging

For this specific project,
I felt like using LLMs would be a good use-case
(as well as test case to see if they improved).
Not only can I feed it specifications,
but I can also set up cross-validation tests
against `htslib` and noodles.
So I can both verify that the code follows the spec as written
(this is what tracey provides)
and also validate the new implementation against real world tools.
On top of this,
I also prioritized adding property tests
and later even fuzz tests.

On top of this,
I also made sure to enable quite strict [`clippy`] lints.
You can see a snapshot of the list [here][lints].
My "favorites" by how annoying/good at preventing issues in seqair
are probably:

1. `arithmetic_side_effects`, which disallows using typical math operators like `+` and `-`,
   because they can overflow/panic.
   Instead, they forcing you to use precise wrapping semantics
   like `.saturating_sub(n)` or `.checked_add(1)`
   and dealing with overflow possibility.
   I think half of what the fuzzer caught, this lint also caught.
2. `indexing_slicing`, which disallows `x[2]`.
   This is another source of panics that are really annoying to catch.
   In some parts of the code we deal with very well known sized arrays, so there we allow this (after assertions!).
   But in general, I prefer `x.get(n).ok_or(SomeError::ThingOutOfBounds)?`[^slicing-perf].
3. `cast_lossless`, `cast_possible_truncation`, `cast_possible_wrap`, and friends.
   They disallow using `x as y` for numeric values where you lose precision accidentally.

[`clippy`]: https://doc.rust-lang.org/1.95.0/clippy/ "800+ lints to catch common mistakes and improve Rust code"
[lints]: https://github.com/Softleif/seqair/blob/3b0e618153c0b192ea669361b23b3b9b5ff9558d/Cargo.toml#L19-L66

[^slicing-perf]: I wonder if LLVM is clever enough to elide this in as many cases as bounds checks.
  And, if it can't, is the code for this case simpler/faster than the panicking slicing?

This does not mean that the code coming out is fantastic,
but it makes me way more confident
than just "vibe coding" something that I am quite serious about.

## A Rustic API

My focus was *also* to make the developer experience very Rust-like,
because that is what I enjoy.
I wanted to have strong types for everything
that enforce logic constraints,
distinguish types with parameters,
and make sure that APIs can only be used in the right order
(using session types/type-state builders[^type-state]).
Let me give an example for each

One example is that some types have zero-based positions
and other (like VCF) count from `1`.
Naturally, I added a type `Pos<T>`
that exists as both `Pos<Zero>` and `Pos<One>`
(aliased to `Pos0` and `Pos1` for convenience).

Another example is that there is one `Reader` entrypoint,
which automatically detects the file format(s)
and provides detailed errors[^logs].
This is not something I've seen `noodles` provide,
and the errors from `htslib` were sadly not very helpful.
Seqair also add [`tracing`] for logs and instrumentation,
but I've not made a lot of use of it.

[`tracing`]: https://docs.rs/tracing/0.1.44/tracing/ "A scoped, structured logging and diagnostics system."

And here's a third example:
When writing VCF/BCF files[^also],
we first need to write a header,
which defines all the fields.
In the binary version, we will refer to them by their ID.
This header also needs to be written in a specific order
(configs, then filters, then info fields, then format fields, then samples).
In the same way,
records (lines) need to be written in a specific order,
so that we can stream them directly to an output buffer.
All of this is enforced using the type-state pattern,
where methods like `VcfHeaderBuilder::formats` or `RecordEncoder::begin_samples`
transition from one state to another,
which then includes different methods to add different data.

[^also]: Yes, I also added this, after [complaining][rastair-post] about how inefficient the Rust wrapper was in my Rastair post.

I think I went further with this than both `noodles` and `rust-htslib`
but I'm not sure this is necessarily better for the average user
who might not be enjoying these Rust features as much as me.

[^type-state]: This is one of my favorite features in Rust that I don't get to use so often.
  [I wrote about it][elegant-apis-in-rust] all the way back in 2016!

[elegant-apis-in-rust]: https://deterministic.space/elegant-apis-in-rust.html#session-types
